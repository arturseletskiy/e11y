# E11y Configuration Design: Devise-Inspired Approach

**Date:** 2026-03-13  
**Status:** Draft (brainstorming)  
**Goal:** Переработать конфигурацию E11y, вдохновившись Devise и best practices гемов

---

## 1. Текущее состояние E11y

**Структура:**
- `E11y.configure { |config| ... }` — блок с объектом Configuration
- `Configuration` — класс с `attr_accessor` и вложенными объектами
- Вложенные конфиги: `rails_instrumentation`, `logger_bridge`, `sidekiq`, `active_job`, `rate_limiting`, `slo_tracking`, `security`, `tracing`, `opentelemetry`, `cardinality_protection`, `error_handling`, `dlq_storage`, `ephemeral_buffer`
- Специальные API: `config.pipeline.use(...)`, `config.adapters[:name] = instance`, `config.rate_limiting { }`, `config.slo { }`

**Проблемы:**
- Глубокая вложенность: `config.rails_instrumentation.custom_mappings`
- Неочевидный порядок настройки (pipeline, adapters, feature flags)
- Генератор создаёт минимальный initializer без документации опций
- Нет единого «справочника» всех опций с дефолтами и комментариями

## 2. Devise: Ключевые паттерны

**Источник:** [lib/devise.rb](https://github.com/heartcombo/devise/blob/main/lib/devise.rb)

### 2.1. mattr_accessor — плоские настройки

```ruby
# Devise
mattr_accessor :stretches
@@stretches = 12

mattr_accessor :remember_for
@@remember_for = 2.weeks

mattr_accessor :timeout_in
@@timeout_in = 30.minutes
```

- Каждая настройка — отдельная переменная на уровне модуля
- Дефолт задаётся при инициализации
- Нет вложенных объектов для простых значений

### 2.2. setup — единая точка входа

```ruby
def self.setup
  yield self
end
```

- `Devise.setup { |config| config.xxx = value }` — config это сам модуль Devise
- Пользователь настраивает напрямую `Devise.stretches = 10`

### 2.3. Специальные блоки для сложной логики

```ruby
def self.warden(&block)
  @@warden_config_blocks << block
end

def self.omniauth(provider, *args)
  config = Devise::OmniAuth::Config.new(provider, args)
  @@omniauth_configs[config.strategy_name.to_sym] = config
end
```

- Warden, OmniAuth — отдельные DSL, не mattr_accessor

### 2.4. Генератор initializer

Devise `rails g devise:install` создаёт `config/initializers/devise.rb` с:
- Всеми опциями закомментированы
- Каждая опция с пояснением и дефолтом
- Группировка по смыслу (mailer, timeout, unlock, etc.)

## 3. Best Practices (по исследованию)

| Паттерн | Описание | Devise | E11y сейчас |
|---------|----------|--------|--------------|
| **Defaults** | Явные дефолты для каждой опции | ✅ mattr_accessor с @@var = default | ✅ в initialize |
| **Single configure block** | Один блок для всей настройки | ✅ setup | ✅ configure |
| **Flat vs nested** | Плоские настройки vs вложенные объекты | Flat (50+ mattr) | Nested (10+ sub-configs) |
| **Generator** | Initializer с полным списком опций | ✅ Все опции в шаблоне | ❌ Минимальный |
| **Validation** | Проверка required/valid values | Частично (в рантайме) | Частично |
| **Documentation** | Комментарии к каждой опции | ✅ В initializer | ❌ В отдельных doc |

## 4. Подходы к редизайну

### Подход A: «Плоский Devise» (mattr_accessor)

**Идея:** Вынести все простые настройки на уровень модуля, как в Devise.

```ruby
# E11y
module E11y
  mattr_accessor :enabled
  @@enabled = true

  mattr_accessor :service_name
  @@service_name = nil

  mattr_accessor :environment
  @@environment = nil

  mattr_accessor :default_retention_period
  @@default_retention_period = 30.days

  def self.setup
    yield self
  end
end

# config/initializers/e11y.rb
E11y.setup do |config|
  config.service_name = "myapp"
  config.environment = Rails.env
end
```

**Плюсы:** Простота, знакомый паттерн, легко документировать  
**Минусы:** adapters, pipeline — не влезают в mattr. Нужны отдельные API. Много переменных на модуле (50+).

---

### Подход B: «Гибрид» (плоские примитивы + блоки для сложного)

**Идея:** Простые настройки — attr_accessor на Configuration. Сложные (pipeline, adapters, rate_limiting) — блоки DSL.

```ruby
E11y.configure do |config|
  # Плоские (как Devise)
  config.service_name = "myapp"
  config.environment = Rails.env
  config.enabled = true

  # Инструментация — плоская с префиксом
  config.rails_instrumentation_enabled = true
  config.sidekiq_enabled = true
  config.active_job_enabled = true
  config.logger_bridge_enabled = false

  # Сложное — блоки
  config.adapters do
    register :logs, E11y::Adapters::Loki.new(...)
  end

  config.pipeline do
    use E11y::Middleware::TrackLatency
    use E11y::Middleware::TraceContext
    # ...
  end
end
```

**Плюсы:** Баланс. Простое — просто. Сложное — выразительно.  
**Минусы:** Два стиля в одном конфиге. `rails_instrumentation` имеет `custom_mappings`, `ignore_events` — не влезает в один bool.

---

### Подход C: «Секции как в Devise initializer»

**Идея:** Не менять API, а улучшить генератор и структуру initializer. Группировать опции в логические секции с комментариями.

```ruby
# config/initializers/e11y.rb (generated by rails g e11y:install)

# =============================================================================
# BASIC
# =============================================================================
E11y.configure do |config|
  config.service_name = ENV["SERVICE_NAME"] || "myapp"
  config.environment = Rails.env
  config.enabled = true
  config.default_retention_period = 30.days
end

# =============================================================================
# ADAPTERS
# =============================================================================
E11y.configure do |config|
  config.adapters[:logs] = E11y::Adapters::Loki.new(url: ENV["LOKI_URL"])
  config.adapters[:errors_tracker] = E11y::Adapters::Sentry.new(dsn: ENV["SENTRY_DSN"])
end

# =============================================================================
# RAILS INTEGRATION (optional)
# =============================================================================
E11y.configure do |config|
  config.rails_instrumentation.enabled = true
  config.sidekiq.enabled = true
  config.active_job.enabled = true
  config.logger_bridge.enabled = false
end

# =============================================================================
# PIPELINE (middleware order)
# =============================================================================
# Default pipeline is configured automatically. Uncomment to customize:
# E11y.configure do |config|
#   config.pipeline.clear
#   config.pipeline.use E11y::Middleware::TrackLatency
#   config.pipeline.use E11y::Middleware::TraceContext
#   # ...
# end
```

**Плюсы:** Минимальные изменения кода. Улучшенный DX за счёт генератора.  
**Минусы:** Не меняет фундамент — вложенность остаётся.

## 5. Рекомендация

**Подход C** — быстрый выигрыш: обновить генератор `e11y:install`, создать шаблон initializer в стиле Devise (секции, комментарии, все опции). Без breaking changes.

**Подход B** — среднесрочно: вынести «feature flags» (rails_instrumentation.enabled, sidekiq.enabled, etc.) в плоские `config.rails_instrumentation_enabled`, `config.sidekiq_enabled`. Оставить блоки для pipeline, adapters, rate_limiting. Потребует миграции.

**Подход A** — радикальный: полный переход на mattr_accessor. Высокий риск, много рефакторинга.

---

## 6. Решения (2026-03-13)

1. **Scope:** Генератор **и** рефакторинг API
2. **Breaking changes:** Да, **без обратной совместимости**
3. **Вложенность:** Переход на плоские `config.logger_bridge_track_severities`
4. **Имя метода:** `E11y.configure` (оставить)

---

## 7. Выбранный подход: B (Гибрид) + полный flatten

- Все примитивные настройки — плоские `config.<section>_<option>`
- Pipeline, adapters — остаются как есть (Hash, Builder)
- Блоки `config.rate_limiting { }`, `config.slo { }`, `config.cardinality_protection { }`, `config.security { }`, `config.tracing { }`, `config.opentelemetry { }` — **удаляются**. Вместо них — плоские accessors + helper-методы где нужна структура (например `config.add_rate_limit_per_event(...)`)
